# Continuity ledger (per-branch)

## Human intent (must not be overwritten)

- When a text generation node has Structured Output configured (`output.format === "object"`), downstream nodes should be able to reference individual fields from the structured output via `@` mentions, rather than receiving the entire JSON blob as a single block.

## Goal (incl. success criteria)

- TextGeneration / ContentGeneration nodes with `output.format === "object"` expose one `Output` per top-level schema property (in addition to the existing full-JSON `generated-text` output).
- Downstream `@` mention suggestion list shows each property as a separate option (`NodeName / propertyName`).
- At runtime, per-property outputs resolve to the extracted field value (stringified for non-string types).
- Existing behavior (full JSON output, `source`, `reasoning` outputs) is preserved.

## Constraints/Assumptions

- Only **top-level** schema properties become individual outputs (nested paths stay as stringified JSON inside their parent property value).
- Already gated by `structuredOutput` feature flag ‚Äî no new flag needed (if `output.format !== "object"`, zero property outputs exist).
- Reuse existing `{{nodeId:outputId}}` mention format ‚Äî no protocol changes needed.
- Reuse existing `GeneratedTextContentOutput` type for per-property outputs ‚Äî no new `GenerationOutput` variant needed.

## Key decisions

- **Accessor convention**: `generated-object-property:<propertyName>` ‚Äî avoids collision with `generated-text`, `source`, `reasoning`; the property name is recoverable by stripping the prefix at generation time.
- **OutputId stability**: When schema changes, diff by property name ‚Äî preserve `OutputId` for unchanged properties so downstream connections/mentions survive schema edits.
- **Connection cleanup**: When a property is removed from the schema, delete connections referencing its `OutputId` before removing the output (same pattern as `model-settings.tsx:110-118` for the `source` output).
- **Value stringification**: Strings pass through; all other types (`number`, `boolean`, `object`, `array`) are `JSON.stringify`'d ‚Äî consistent with how prompt interpolation expects string values.
- **Atomic update**: Use `updateNodeData(node, { content, outputs })` (single call) rather than separate content/outputs updates to avoid intermediate inconsistent state.
- **Failure handling**: If `JSON.parse(text)` fails at generation time, skip property outputs silently ‚Äî the full JSON output still holds the raw text.
